| ID | Severidade | Área | Achado | Evidência |
| --- | --- | --- | --- | --- |
| C-001 | **P0** | `server.ts` `/api/engine/*` | `new ExecutionEngine(orchestrator)` é criado **sem `kernel`, sem `eventStore`, sem `stateStore`**. Isso significa: (a) o `PolicyEngine` avalia com o fallback fixo `kernelMode: "SAFE_WRITE"` — o modo real do kernel (inclusive `LOCKED`) é ignorado por essa rota; (b) nenhum evento operacional é gravado; (c) nada é persistido em `ExecutionStateStore`. | `src/server.ts:28` (`const engine = new ExecutionEngine(orchestrator);`) comparado com `src/index.ts` que agora passa `(orc, undefined, undefined, kernel, eventStore, stateStore)` |

## 3. O que está saudável (não mexer sem necessidade)

- CLI (`run`, `run status`, `run abort`, `run approve-step/reject-step`) usa o Engine corretamente, com `kernel`, `eventStore` e `stateStore` sempre passados.
- `PolicyEngine` default chain (`SecretsPolicy → ModePolicy → StepTypePolicy → SafetyPolicy → GitPolicy → DeployPolicy → ScopePolicy → ShellPolicy`) é avaliada em toda chamada de `runOnce()` vinda do CLI, sem exceção.
- `ShellExecutor` continua fora do registry padrão; `ShellPolicy` exige opt-in explícito (env) **e** `EXECUTE_APPROVED`; `CommandValidator` restringe a uma whitelist read-only. Três camadas independentes — nenhuma sozinha libera execução real.
- `ClaudeExecutor`/`OpenCodeExecutor` são stubs puros: nenhuma chamada de rede, nenhum uso de env/secret, retorno sempre determinístico.
- `ExecutionStateStore` e `EventStore` (quando usados via CLI) refletem progresso, erro e abort de forma consistente com o que os testes verificam.

## 4. Plano de integração real (Claude/OpenCode) — proposto, não executado

**Pré-requisito:** resolver C-001/C-002/C-003 antes de qualquer chamada externa real ser plugada, porque hoje o `server.ts` daria a um cliente HTTP (sem autenticação, escutando em todas as interfaces — B-004 da auditoria anterior) um caminho para disparar execução real ignorando o `PolicyEngine`.

Passos sugeridos, na ordem:

1. **Config por env, no padrão do `llm-config.ts`** — criar `execution/executor-config.ts` com `JOEFELIPE_CLAUDE_EXECUTOR_ENABLED`, `JOEFELIPE_OPENCODE_EXECUTOR_ENABLED` (default `false`/mock), espelhando como `loadLlmConfig()` decide entre `mock` e `openrouter`.
2. **`ClaudeExecutor`/`OpenCodeExecutor` ganham um modo real opcional** atrás dessa flag — quando desligada (default), comportamento atual (stub, sucesso simulado) é preservado byte a byte (não quebra os 139 testes existentes).
3. **Nenhuma chamada real deve ocorrer se `PolicyEngine` não aprovar** — o desenho atual já garante isso (o engine só chama `executor.execute()` depois do `verdict.allowed`), então não deveria exigir mudança estrutural, só a implementação real dentro do executor.
4. **Testes:** o modo real precisa de um teste com `execFn`/`fetchFn` injetado (mesmo padrão do `ShellExecutor`/`OpenRouterProvider`), nunca uma chamada de rede de verdade em CI.
5. **Corrigir C-001/C-002 primeiro** (ver seção 5) para que o `server.ts` não vire uma porta lateral que ignora tudo isso.

## 5. Plano para os demais pendentes (ordem sugerida)

1. **C-001/C-002/C-003 (server.ts)** — passar `kernel`, `eventStore`, `stateStore` para o `ExecutionEngine` do server (igual ao CLI); mover `/api/orchestrator/complete|fail` para passar pelo Engine (ou documentar explicitamente por que essas rotas de baixo nível continuam existindo, com uma flag de habilitação separada). Isso por si só já resolve boa parte do item 4 do pedido do usuário ("expor status/approval no server.ts") porque exige tocar as mesmas rotas.
2. **C-004** — adicionar `/api/execution/state` (le `ExecutionStateStore`) e `/api/execution/approve-step/:id` + `/reject-step/:id`, reaproveitando `TaskOrchestrator.approveStepHuman/rejectStepHuman` (sem duplicar lógica).
3. **C-009 (cancelamento durante executor longo)** — propagar um `AbortSignal` construído a partir do `ExecutionStateStore`/flag em memória para dentro de `ExecutionCommand` (campo novo opcional, não quebra o tipo atual) e os executores reais (quando existirem) devem respeitá-lo.
4. **C-007 (unificar approval)** — decisão de arquitetura pendente: ou (a) manter os dois níveis mas deixar explícito na UI/CLI que são coisas diferentes (missão vs step), ou (b) fazer `ApprovalManager` também saber sobre steps `waiting_human` (índice cruzado por `missionId`). Recomendo (a) por enquanto — menor risco, e o `run status` já deixa isso visível.
5. **C-008 (rotação do EventStore)** — validar com um teste de carga leve (gerar >10MB de eventos sintéticos em teste, confirmar que `rotateIfNeeded()` dispara e `pruneRotated()` mantém o teto de 20 arquivos). Não decisão de produto, só confirmação de que a proteção existente funciona sob volume.
6. **C-005/C-006 (tech debt)** — trocar o `Record<string,string>` implícito por um campo tipado (`pending?: boolean` em `StepResult`, ou um novo campo `signal?: "pending_human"`) e passar a lançar/capturar `PolicyDeniedError`/`ApprovalRequiredError` de fato no engine, para o "padrão de erro" declarado bater com o que é usado.

---

## 6. Recomendação

Concordo com a proposta do usuário: **não integrar modelo real hoje**. A prioridade imediata, antes de qualquer integração externa, é **C-001/C-002/C-003** — não porque o Core novo tenha um bug, mas porque a superfície HTTP pré-existente (`server.ts`) não foi atualizada para usar o Core novo e hoje contorna o PolicyEngine inteiro. Fazer a integração real de Claude/OpenCode antes de fechar isso significaria abrir uma chamada externa de verdade atrás de uma porta que já sabemos que vaza.

**Próxima missão segura recomendada:** corrigir C-001/C-002/C-003 (wiring do `server.ts` com kernel/eventStore/stateStore + mover `complete/fail` para passar pelo Engine), com testes cobrindo que uma requisição HTTP `/api/engine/run` respeita o modo do kernel do mesmo jeito que a CLI respeita. Só depois disso faz sentido plugar `ClaudeExecutor`/`OpenCodeExecutor` reais.
